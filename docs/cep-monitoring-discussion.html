From investigation to standing monitor

A CEP / impact-analysis layer: a scheduled job ingests fresh news, extracts events & actors, intersects them with the global knowledge graph, and propagates impact onto connected nodes — direct or hidden.

Draft for review — nothing here is built yet. Numbers and names are illustrative.

The shift: pull → push

Investigator today is request-driven: you start an investigation and it builds a graph. This turns it into a standing monitor: it watches the world and tells you when something in your graph moves.

What it reuses vs. what's new

The concept maps onto pieces we already have — roughly 80% reuse.

reuses

  • Event & actor extraction (dspy + gpt-4.1)
  • Canonicalization — new names → KG entities
  • Cumulative KG + structured store = the global graph
  • Connector — direct & hidden reach, brokers
  • Junction-tree belief propagation — fuse evidence, propagate to neighbours
  • Search sources — news / Wikipedia / GDELT

new

  • A scheduler / standing job runner
  • A watchlist scoping primitive
  • An intersection filter (KG-relevant news only)
  • An impact model + scorer
  • A temporal layer (event-time, decay, timeline)
  • A digest / alert surface

The daily pipeline

The crux: what is “impact”?

This is the make-or-break choice. Three framings — I'd combine the first two.

FramingIdeaReuses
Belief-shift
principled core		 A new event is fresh evidence on entity X → re-run belief propagation → the posterior deltas across the graph are the impact, flowing to neighbours by construction. Junction-tree BP
Topological reach
the “direct or hidden”		 Use the connector to find X's affected neighbourhood — direct and brokered/indirect — weighted by path proximity & strength. Connector / hidden-paths
Relevance gate
noise control		 Only surface impact on nodes the user actually watches. Watchlist
My lean: connector to scope the affected subgraph (direct + brokered) → belief propagation on that local subgraph for principled posterior deltas → relevance-rank for the digest.

The two honest hard parts

1 · Temporal data exists per-investigation, but the global graph strips it. We already extract solid temporal structure: event dates (ISO-8601 on event nodes), per-actor timeline_events (dated mini-timelines), and real event→event edges (event_followed_by / event_coincident from shared participants + date ordering), plus article published_date and a timeline filter. The gap is narrower than “no model”:
  • The cumulative KG drops event nodes & temporal edges on merge (the headline-node fix), so the global graph monitoring intersects against is currently atemporal — actors + relationships only.
  • Actor↔actor relationships have no validity interval (when a link started/ended) and no decay (old evidence weighs like new).
So the work is mostly: carry the existing event-time / timeline data into the global graph (a temporal sidecar, like we did for structured props) + add validity & decay — not inventing temporal extraction.
Agreed direction (independent of the CEP decision): update the global graph model to preserve the temporal data — event dates, per-entity timeline_events, and the event→event ordering edges — so the cumulative KG carries a real timeline per entity. This is worth doing regardless of whether we build full monitoring.
2 · BP needs a chordal / TMFG graph; the global KG isn't one. Either build a small local TMFG around each touched entity per event (cheap, scoped — keeps our exact BP), or swap to a diffusion / personalized-PageRank propagation. The local-TMFG route is my lean.

Plus the usual: daily top-k news is mostly noise (the intersection filter is essential); false entity-matches create phantom impacts (matching must be careful); cost & drift grow as the KG grows daily.

An MVP shape

1 · Watchlist

A set of entities (or a domain) the user cares about — the scoping primitive, not “all the KG.”

2 · Daily job

Top-k news + search sources → extract → canonicalize → keep only events that intersect the watchlist / KG.

3 · Impact

Per event: connector-scope the affected subgraph; score by event strength × proximity × watch-relevance; optional local-TMFG BP for deltas.

4 · Digest

A dated impact digest — new events, entities hit, ripple to connected nodes, any new hidden connections formed — ranked, with an alert threshold.

5 · Temporal

Stamp every event/impact so each entity accrues a timeline you can scrub.

Phasing

PHASE 1 · achievable now

Scheduled impact digest

Daily intake → intersect → impact (connector + local BP) → ranked digest with thresholds. Reuses the existing stack.

PHASE 2 · ambitious

CEP pattern rules

Detect patterns of events over time, e.g. “A sanctioned → B linked to A → C transacts with B within 30 days” → alert. Needs the temporal layer first.

Three questions to align

  1. Who's the user / what's the watchlist? A risk analyst watching a portfolio of entities? A journalist tracking one story's network? This decides whether monitoring is scoped per-watchlist or runs over the whole KG.
  2. Impact = belief-shift, topological reach, or both? (I lean both — connector to scope, BP to score.)
  3. Phase-1 scope: ship the impact digest first with CEP pattern rules as a later phase — or do you want pattern-detection in the core from the start?
